Q3. What is the difference between GNU and GNUSH?

A3.    Very little - GNUSH is a version of the FSF GNU sources that has been prebuilt for your convenience.
The GNU Project was launched to develop a complete Unix-like operating system which is free software: the GNU system. GNU is a recursive acronym for "GNU's Not Unix" (it is pronounced "guh-NEW".)  The GNU system per se is not yet finished, but its offshoot GNU/Linux (which uses Linux as the kernel) is widely used.  See http://www.gnu.org/ for more information about GNU.

 

A subset of GNU is its development tools, which include command-line-basedcompiler, linker and various binary utilities. The source code for these tools,while freely available, generally does not include pre-built binaries. It has not been subjected to formal testing nor is there a guaranteed mechanism for new platform support. That solution is suitable to some developers, either because of cost constraints or because they want to evaluate GNU before embarking on anew software development project.

 

We endeavor to provide a readymade GNUSH tool chain based on the GNU standard for Renesas(formerly Hitachi and Mitsubishi) SH series of microprocessors / micro-controllers on Windows and on GNU/Linux. This tool chain is built on a standard reference base and delivered with pre-configured binaries. The GNUSH tool chain is subjected to thorough testing before it is released.  The advantage is that the embedded developers need not spend time in building and supporting the development tools and can concentrate on application development

    Q8. What is the difference between Cygwin and MingW ?

    A8. CygWin provides a transparent Unix emulation layer on Windows. This makes it possible to rebuild large, existing Unix applications with minimal changes to their source code, using GCC. The disadvantage is a substantial decrease in speed for both the compiler, and the windows applications built with it because of the overhead introduced by the Unix emulation layer (cygwin.dll)

    The Minimalist Gnu-Win32 project (MingW32) was started by Colin Peters. Instead of emulating Unix, it uses Microsoft's runtime DLLs(either CRTDLL.DLL or MSVCRT.DLL).

    MingW also contains the w32api package, which is a set of includes and import libraries to enable the use of Windows32 API.

    Q.11 When I generate .out file and load it into GDB, the initialization data does not appear in the .mdata section. Do I need to define the macro ROMSTART?
    OR
    When I make a ROM image, the .data section is not copied in ROM with the _mdata label, but only in RAM with the _data label. How can I get it copied to _mdata section?

    A.11 Every loadable or allocatable output section has two addresses. The first is the VMA, or virtual memory address. This is the address the section will have when the output file is run. The second is the LMA, or load memory address. This is the address at which the section will be loaded. In most cases the two addresses will be the same. An example of when they might be different is when a data section is loaded into ROM, and then copied into RAM when the program starts up (this technique is often used to initialize global variables in a ROM based system). In this case the ROM address would be the LMA, and the RAM address would be the VMA. The macro ROMSTART is used to copy initialized variables loaded at _mdata (LMA) to _data (VMA) at startup.

    If you are using debugger to load object file then ROMSTART should not be defined. The loader in debugger (GDB/HEW/HDI) loads the content at VMA so you don't have to copy it in the startup routine.

    If you are NOT using the debugger then make sure you have the following statements in the linker script
    .data : AT (_mdata) {
    _data = .;
    *(.data)
    _edata = .;
    } > ram

    This will load initialized variables at _mdata section in ROM.

    If you want to debug the code copying data in startup routine and are using debugger then loading S-Record file first and then COFF or ELF file will solve the problem.

    If you are using HEW, in Release configuration above thing is done by default. If you want to do it in Debug configuration, go to Options -> Linker -> Sections .data. Then in Modify section group -> Advanced, check "Reserve memory for section mapping" box and add label as _mdata.
    Upto v0702, HEW "Debug" and "Release" configurations are different. In "Debug", the initialized data is loaded at the VMA (in RAM) whereas in "Release" the initialized data is loaded at the LMA (in ROM) and the macro ROMSTART is used to copy data from ROM to the RAM addresses as explained above.

    However, from v0703 onwards, HEW "Debug" and "Release" configurations have been made identical so that the initialized data will always be placed in the ROM (LMA) and then copied from ROM to RAM (VMA) at start-up. The HEW loader has been modified to download sections at their LMA instead of VMA and the macro ROMSTART has been removed.

    Q12. I get undefined reference to library calls even when I have specified the libraries and path for libraries correctly.
    E.g. I get the error "undefined reference to functions like ___mulsi3, ___umodsi3", even when I have specified "-lc" or "-lg" and their paths.

    A12. Please note that linker needs the object files and libraries to be provided in particular sequence. You need to specify all the object files prior to library files when specifying the library path explicitly on command line.
    i.e. The command format should be as shown below:

    *-*-ld <LD_OPTIONS> <OBJECT_FILES> -o <OUTPUT_FILENAME> -T<LINKER_SCRIPT> -L<LIB_PATH> <LIBRARIES>

    E.g. sh-elf-ld -EL 1.obj 2.obj -o fin.out -Tlinker.lnk -L./libpath/ -lc -lg -lgcc

    Otherwise, you may use an option to the linker "--start-group <archives> --end-group" to get rid of specifying the libraries in particular sequence. By grouping the archives, they all are searched repeatedly until all possible references are resolved.

    Refer to following link for details, http://www.kpitgnutools.com/manuals/ld.html#IDX120

    Q13. How to use static analysis feature over the command line?

    A13. Static analysis feature can be invoked on command prompt using option '-Wanalyze'. This option will analyze application code for different set of rules and prompt warnings. These warnings will be prompted against violation of rules. You can specifically enable warning for a particular rule using command line option '-Wanalyze-rule='. Also, warning for a particular rule can be disabled using command line option '-Wno-analyze-rule='. However, this option should be preceeded by '-Wanalyze' option. For more information on the same, please refer to the GCC user manual.
    For Eg:
    1. If you want to analyze code for all rules, then use '-Wanalyze' option.
    2. If you want to analyze code for only rule 1, then use '-Wanalyze-rule=1' option.
    3. If you want to analyze code for all rules except rule 1, then use '-Wanalyze' option followed by '-Wno-analyze-rule=1' option.

    Q14. What is binary utilty 'convrenesaslib'? What is its functionality? OR Can I convert Renesas SH files to GNU archive?

    A14. The GNU 'convrenesaslib' utility converts Renesas SH library file(s) into GNU archive(s). The converted GNU archive has '.rlib' extension.
    The 'convrenesaslib' utility extracts the object files from the input Renesas SH library and appends the filenames with '.robj' extension. The 'ar' utility is then invoked to create a GNU archive of these extracted object files. The GNU archive created will have '.rlib' extension appended to the name. These new extensions are necessary so as to distinguish from other files with same names. The extracted object files are deleted after the creation of the GNU archive.

    This utility supports only Renesas SH library files. For details on the usage of this utility, please refer to the binutils manual on kpitgnutools website: www.kpitgnutools.com/manuals/binutils.html#SEC140

    Q15. I get 'address error' while trying to execute the 'fmov.d' instruction for SH2A target. Is it a known problem?

    A15. Yes, this is a known problem.

    GCC generates fmov.d instruction for SH2A target by default. This instruction transfers the double values from memory to register OR from register to memory. The SH2A target expects the double longword data to be accessed from double longword boundary (i.e. 8-byte aligned memory).

    The instruction "fmov.d" will generate "address error" exception when the memory address from where the data fetched is not 8-byte aligned. Currently, the GNUSH compiler does not always ensure that the memory address is 8-byte aligned for "fmov.d" instruction.

    We intend to fix this known problem in future releases.

    Q17. For toolchain versions v0903 onwards, why do I get multiple definition errors for functions in syscalls.c? E.g. I get the error "multiple definition of '_link'". I do not get this error for toolchain version v0902 or lower.

    A17. Toolchain versions v0903 and onwards, require the function name "isatty" in "syscalls.c" to be prefixed by a '_' character.

    Following is the implementation of the "isatty" function:
    -------------------------------------------
    isatty (fd)
    int fd;
    {
    return 1;
    }
    -------------------------------------------
    The above implementation needs to be changed as follows (for toolchain versions v0903 and above):
    -------------------------------------------
    _isatty (fd)
    int fd;
    {
    return 1;
    }
    -------------------------------------------

    -mbigtable

    -mfmovd

    Enable the use of the instruction fmovd.

    -mhitachi

    Comply with the calling conventions defined by Renesas(formerly Hitachi).

    Mark the MAC register as call-clobbered, even if `-mhitachi' is given.

    Increase IEEE-compliance of floating-point code.

    Dump instruction size and location in the assembly code.

    -mprefergot

    -musermode

    Generate a library function call to invalidate instruction cache entries, after fixing up a trampoline. This library function call doesn't assume it can write to the whole memory address space. This is the default when the target is sh-*-linux.

    Q1. How do I start writing a linker script?

    A1.    Following section explains the linker script commands for C,C++ programs in brief-

    The Entry Point

    ENTRY(symbol)

    First executable instruction in an output file e.g. ENTRY(_start)

             

    Memory Layout

    MEMORY

    {

                 name (attr) :ORIGIN = origin, LENGTH = len

                ...

    }

    The `MEMORY' command in linker script describes the location and size of blocks of memory in the target. 

     

    name- is a name used internally by the linker to refer to the region.

     

               (attr)- is an optional list of attributes

    e.g.`R' Read-only sections.

                    ‘W'  Read/write sections.

          `X' Sections containing executable code.

                  ...

    origin- is the start address of the region in physical memory.

    len-  is the size in bytes of the region

    e.g.

    MEMORY

    {

      ram (rw) : o = 0x0400000, l =512k

      rom (rx) : o = 0x0000000, l =512k

    }

    Section Definitions

    SECTIONS{ ...  secname : {    contents  }... }secnameis the name of the output section, and contents a specification of what goes there e.g. a list of input files or sections of input files

    Section Placement

    In a section definition, the contents of an output section can be specified by listing particular input files, by listing particular input-file sections, or by a combination of the two. An arbitrary data can also be placed in the section,and symbols can be defined relative to the beginning of the section.

    The section definition may include any of the following kinds of statement.

    filename

    e.g..data : { afile.o bfile.o cfile.o }

    filename(section )

    filename(section , section, ... )

    filename(section section ... )

    *(section)

    *(section, section, ...)

    *(section section ...)

     

    The Location Counter

    The special linker variable ‘.’ always contains the current output location counter.

     

    For the GNU C++ compiler, set up constructor and destructor tables. Set up the .tors section as follows:

    .tors:


    {

        __CTOR_LIST__ =.;

        LONG((__CTOR_END__ -__CTOR_LIST__) / 4 - 2)

        ___ctors = .;

       *(.ctors)

        ___ctors_end = .;

       LONG(0)

       __CTOR_END__ =.;

        __DTOR_LIST__ =.;

       LONG((__DTOR_END__ -__DTOR_LIST__) / 4 - 2)

        ___dtors = .;

       *(.dtors)

        ___dtors_end = .;

       LONG(0)

        __DTOR_END__ =.;

    } > rom


    The symbol __CTOR_LIST__marks the start of the global constructors and __CTOR_END__marks the end. Similarly the symbol __DTOR_LIST__ marks the start of the global destructors and __DTOR_END__ marks the end.The first word in the list is the number of entries, followed by the address of each constructor or destructor,followed by a zero word. GNU C++ will normally arrange to put the addresses of global constructors and destructors into the .ctorsand .dtorssections.

    The special section name ‘/DISCARD/’ may be used to discard input sections. 

    The comments in the linker script can be included just as in C, delimited by ‘/*’ and ‘*/’.

     The linker script also contains expressions.

    Expressions may inturn include integers, symbol names, operators, arithmetic functions.

               Some of the Section Data Expressions are –

               CREATE_OBJECT_SYMBOLS

                Create a symbol for each input file in the current section, set to the address of the first byte of data written from that input file.

               BYTE(expression)

                           Place one byte at the current address of that section.

               FILL(expression)

                           Specify the "fill pattern" for the current section.

               etc.

               Some of the Optional Section Attributes are -

           Start

    The output section can be forced to be loaded at a specified address.

               (NOLOAD)

               This directive will mark a section to not be loaded at run time.

               AT ( ldadr )

               ldadr specifies the load address of the section.

                >region

    Assign this section to a previously defined region of memory (e.g.
    ram,rom). etc.

     

    Option Commands

    The linker script may include a number of other commands that you can use for specialized purposes.

    OUTPUT_ARCH(bfdname)-

               Particular output machine architecture e.g.OUTPUT_ARCH(sh)

               STARTUP ( filename )

    Ensure that filename is the first input file used in the link process.

               Etc.

     

    SECTIONS - .text (code), .data (initialized data), .bss (uninitialized data). These 3 sections are present for all three object file formats -ELF, COFF, HMS.

    For ELF object file format you need to specify sections .init (invoke global constructors), .fini (invoke global destructors), .got (Global Offset Table),.rodata (Read Only Data) other than the 3 sections mentioned above.

     

    An example of .stack section definition :

    .stack0x043FC00 :

    {

       _stack = .;

       *(.stack)

    }>ram

     

    Depending upon the requirement of target the addresses of stack and text sections are to be specified.

     

    For the GNU C++ compiler, .gcc_exc section may also be needed for COFF and ELF if you get a warning for this section. This is related to the location of the main exception table for constructors.

    The .eh_frame section contains pointers that deal with exceptions.

    Q2. What is a 'specs' file?

    A2. The ‘specs’ file contains the default options to be passed to the assembler/compiler/linker etc. Every GCC tool chain contains a specs file, named ‘specs’. This file is present in \lib\gcc-lib\\ directory.

    $ sh-elf-gcc -v

    The first line of the output displays ‘Reading specs from ...’

    The ‘specs’ file can be modified.

    Alternatively the builtin specifications can be overriden using -specs=’filename’ command line option. Multiple specification files can be specified on the command line and are processed in order from left to right.

    A lot of customisation can be done using the specs file. These changes are global and affect all users of the system.

    Further details about ‘specs’ language can be found in the source file gcc.c under heading ‘The Specs Language’.

    Q8. How can I customize low level functions printf, scanf for my project? OR How can I divert the standard output to UART for my project? OR How can I read the standard input from UART for my project?

    A8. Functions like printf,scanf internally calls low level functions write(), read() respectively. Low level routine write() is used by the printf to send the bytes to the output device. read() function is used by the scanf to read the bytes from the output device.

    To divert the characters towards UART you will have to modify write() function.

    This routine is available in syscalls.c file. This file is available in NEWLIB sources/newlib/libc/sys/sh. Once you change syscalls.c and link the file to your project, printf will call write() automatically.

    For e.g.

    int _write(file, ptr, len)

    int file;

    char *ptr;

    int len;

    {
    /* Your implementation to send the character to the serial port.*/

    // Call sci0_putc()

    return len;
    }

    To read the characters from SCI you will have to modify read() function.

    This routine is available in syscalls.c file. This file is available in NEWLIB sources/newlib/libc/sys/sh. Once you change syscalls.c and link the file to your project, scanf will call read() automatically.

    For e.g.

    int _read (int file,

    char *ptr,

    int len)

    {
    /* Your implementation to receive the character from the serial port.*/

    // Call sci0_getc();

    return len;
    }

    _read function reads only one character at a time therefore "len" is always 1.

    Q17. Why does access to some peripheral registers performed through structures not work on the SH hardware?

    A17. The SH peripheral registers have strict access size requirements with some registers requiring a byte level access, while some requiring word level access.

    The GNUSH compiler generates a word level instruction like "mov.w" to access peripheral registers that require byte level access. This type of access currently fails on the hardware. This behaviour is observed on the v0802, v0803, v0901 ,v0902, v0903 versions of the GNUSH toolchain. Toolchain versions v0801 and older give expected results. Toolchain version v10.01 on wards, the peripheral registers will be accesses using the strict access requirements as long as the 'volatile' keyword is used in their declarations.

    The following workaround is suggested for user using toolchain version v0802 to v0903.

    As a workaround, you can declare two structures. One structure should be declared with the attribute 'packed' while the other structure should not have any attributes. Use the 'packed' structure for peripheral registers requiring byte access and the regular structure for word and long level access.

    E.g.,
    1. Declare the structure using the packed attribute

    2. Add an '_b' at the end of the name to identify byte level access struct __attribute__ ((packed)) st_cpg_b { // Your structure elements : No modification to elements is required };

    3. Add a '_w' to the original declaration of the structure to identify the word level access struct st_cpg_w { // Your structure elements : No modification to elements is required };

    4. Define your structure instances, CPG_B for byte level access and CPG_W for word and long level access #define CPG_B (*(volatile struct st_cpg_b *)0xFFFE0010)/* CPG_B Address*/ #define CPG_W (*(volatile struct st_cpg_w *)0xFFFE0010)/* CPG_W Address*/

    You may then access peripheral registers that require byte level (like register ACLKCR on the SH7211 device) access using the CPG_B structure,

    CPG_B.ACLKCR.BIT.ASDIVS=1; // Performs byte level operation like mov.b

    You may then access peripheral registers that require word level (like register FRQCR on the SH7211 device) access using the CPG_W structure,

    CPG_W.FRQCR.BIT.RNGS = 1; // Performs word level operation like mov.w

    The long data type access should also use the CPG_W structure for its access.

    Q 18. How can I place variables (strings, arrays etc) and const data types into a memory at a fixed address? How can I access them from C code? Can I also declare a function at a pre-fixed address ? Example: I want to set a function at address 0x400.

    steps given below,
    1. Consider the name of the section as "my_data_section" to place the data variables at an address 0x400 and "my_function_section" to place the functions at an address 0xA000.
    Consider the following variables to be placed at some fixed location,
    int iMyInt = 9;
    float fMyFloat = 2.5;
    char cMyArray[]=;
    Define the two sections in the linker script as follows:
    .my_data_section 0x0400 :
    {
    /* */
    _my_data_section = .;
    *(.my_data_section)
    _emy_data_section = .;
    }
    .my_function_section 0xA000:
    {
    /* */
    _my_function_section = .;
    *(.my_function_section )
    _emy_function_section= .;
    }
    The linker script is used to place the sections at the required address/location in the memory.

    2. Define the variables and functions using an attribute as follows in your 'C' code:
    int iMyInt __attribute__ ((section (".my_data_section "))) = 9;
    float fMyFloat __attribute__ ((section (".my_data_section "))) = 2.5;
    char cMyArray[] __attribute__ ((section (".my_data_section ")))="abcde" ;
    int iMyArray[5] __attribute__ ((section (".my_data_section ")))={1,2,3,4,5} ;
    double MyDouble __attribute__ ((section (".my_data_section "))) ; //consider uninitialised variable
    Constant variable can be placed in a specific section as follows:
    const int ciMyConstantInt __attribute__ ((section (".my_const_section"))) = 100;

    where the ".my_const_section" section can be defined in the linker script similarly as above. Please note, by default, the constant variables are allocated in the ".rodata" section.
    Function can be assigned to a particular section as below,
    void my_func_foo(void) __attribute__ ((section (".my_function_section")));

    By default, the compiler places the code that it generates in the ".text" section.
    In the above example, the function "my_func_foo" will be placed in the user defined
    section ".my_function_section", at an address of 0xA000 specified in the linker
    script in step 1..

    Q 20. I am using optlib libraries. The default heap implementation, requires to allocate the heap between _end symbol and the stack pointer. What if I want the heap to be allocated at some other separate memory region?

    A 20: Heap is the memory area where global allocation takes place. It starts after the _end symbol (at the end of .bss section) in the linker script and ends at the current stack pointer. Thus the location of the heap can be modified by changing the _end symbol in the linker script.
    The default implementation for heap in optlib is as mentioned above.

    The following changes have to be done to relocate the heap at a separate memory region,
    1. Add a new section for heap in linker script having the required memory address e.g.,
    2. Delete the current definition of _end symbol from the linker script. Add _end symbol in this new section or alternatively define a new symbol at the start of this new section, e.g. _heap_start.
    The new section will look as shown below,
    .heap 0xaddress :
    {
    __heap_start = .;
    __heap_end = . + 0x800;
    }

    3. As seen above the symbol _heap_end is defined at memory location 0x800 bytes after _heap_start (__heap_end = __heap_start + ).
    This symbol will be used to indicate the end of the heap location. In this case, the heap size will be 0x800 bytes. The user can change the heap size as per their requirement.

    4. The following code has to be included in the user project.
    a. Create a new "c" file having the name newheap.c and paste the following code in it. This will override the default implementation of heap in optlib.
    /////////////// newheap.c /////////////////
    #include
    extern char _heap_end;

    char* _heap_of_memory = NULL;

    char* _last_heap_object = NULL;

    char* _top_of_heap(void)
    {
    return (&_heap_end);
    }
    //////////////End of newheap.c///////////////

    b. The following initialisation function can be place in any file in the project and should be called before using heap area.
    void heap_initialise(void)
    {
    _heap_of_memory = (char*) &_heap_start;
    _last_heap_object = (char*) &_heap_start;
    return;
    }

    Please note that the optlib does not implement sbrk() function.

    Q2. What is a patch and how do I apply it ?

    A2. Patch is an output of diff command, which is to be applied to original file.

    Synopsis of diff is-

    diff [originalfile] [changedfile]

    The simplest way of applying a patch is to keep the original file and patchfile in the same directory.

    The synopsis of patch utility is -

    patch [options] [originalfile [patchfile]]

    but usually just give

    patch -pnum

    Where pnum means strip the smallest prefix containing num leading slashes from each file name found in the patch file. A sequence of one or more adjacent slashes is counted as a single slash.

    'Patch' normally replaces each file, file, with its new version, saving the original file in 'file.orig', and putting rejects hunks (if any) into 'file.rej'.

    If the patch is reversed or already applied you get the following message -

    1. Reversed (or previously applied) patch detected!

    Assume -R? [n]

    Apply anyway? [n]

    Result - This skips the patch.

    2. Reversed (or previously applied) patch detected!

    Assume -R? [n]

    Apply anyway? [n] y

    Result - The patch application fails.

    3. Assume -R? [n] y

    Result - The patch application succeeds.

    When 'y' is pressed, patch then attempts to
    swap each hunk around before applying it.

    For more information do

    $ man patch

    Q5. How do I build GMP and MPFR libraries?

    A.5

    The MPFR library is a C library for multiple-precision floating-point computations with correct rounding. GMP is a free library for arbitrary precision arithmetic, operating on signed integers, rational numbers, and floating point numbers. The GMP (version 4.1 or later) and MPFR (version 2.3.0 or later) libraries are required while building the toolchains and building target applications for GCC (version 4.3.0 or later).

    Execute the following steps to build GMP for Linux toolchain:

    1. Create three directories, gmp_source, gmp_linux_build and gmp_linux_prefix in the home directory.

    2. Copy the GMP source in the 'gmp_source' directory. You can download the GMP source from the following link, http://gmplib.org/#DOWNLOAD

    3. Change the current directory to 'gmp_linux_build' directory.

    4. Execute the following commands,
    i. /home/gmp_source/configure --disable-shared --prefix=/home/gmp_linux_prefix
    ii. make
    iii. make check
    iv. make install
    This will install GMP libraries in '/home/gmp_linux_prefix' directory.

    Q8. Are there any problems, if my host operating system is Windows 9X?

    This may cause problem, if any of the tool chain is installed in the directory which when accessed through command line exceeds the length limit.

    2. In HEW (High-performance Embedded Workshop), this restriction may cause problem during linking. Please try the following steps in case you face the problem:

    a. Install HEW on the root directory.
    b. Run the linker command-line through the batch file.

    To perform this:
    - Disable the linker build phase for the project.
    - Build the project. This will compile the sources.
    - Create a batch file of the linker command line and save to the configuration directory.
    - Export the linker sections to a file and save the file to the configuration directory.
    - Remove the paths from the object files and add thelinker section file using -T option.
    - In the command prompt, run the batch file from configuration directory.
