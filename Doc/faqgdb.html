Q1. What is GDBSH and what does it consist of?

A1.    GDBSH is a cross debugger required for debugging embedded system applications to run on Renesas SH series of targets.

          GDBSH debugger comprises of the following:


Cross Platform Debuggers, Simulators:


Utilities

Description

sh-elf-gdb     

Debugger

sh-elf-run            

Simulator

          The GDBSH on Windows comes with Insight, a graphical user interface to GDB.

          Note: sh-elf-gdb can be used to debug coff objects built by sh-coff toolchain.

Q2. How do I debug my programs using gdb?

A2.To debug your programs using gdb, the GDB Stub for the target is required. GDB stub is necessary to communicate with the target. Use the -g option while building the application to generate the debug information.

The gdb is provided in a separate installer named as GDBSH debugger/simulator and is no longer provided along with the GNUSH tool chain to keep the download size minimum.

The details of the GDB protocols are available at:
http://www.embedded.com/1999/9911/9911feat3.htm

A sample stub for SH2 is available at:
http://sourceforge.net/projects/gdbstubs/

Either the information on these links or the sh-stub.c in the GDB sources can be referred.

You can also simulate the target in GDB. For the simulated target the gdb stub is not required.

Q3. How do I run a program using the SH simulators?

A3. The SH simulators are sh-*-run.exe on windows and sh-*-run on GNU/Linux.

Consider a small program:

int main(void)
{
printf("Hello World!");
}

Compile the program as follows-

$sh-*-gcc -o hello.out hello.c

This will create final binary executable file, hello.out. Run the hello.out file using simulator as follows-

$sh-*-run hello.out

This will print the "Hello World!" string on console.

Q4. What is a '.gdbinit' file?

A4. When GDB is started, it automatically executes commands from its init file, '.gdbinit' on Unix and `gdb.ini' on DOS/Windows. During startup, GDB does the following:

1. Reads the init file (if any) in your home directory.

2. Processes command line options and operands.

3. Reads the init file (if any) in the current working directory.

4. Reads command files specified by the `-x' option.

Init files are not executed if you use `-nx' as the command line option.

Q5. How do I set/change the architecture in gdb?

A5. To set/change the architecture in gdb, open the console window from the GDB and use following command:

set architecture sh2

You may include this command "set architecture sh2" in gdb init file ".gdbinit".

Q1. What should be done if gdb simulator doesn't load global variables while running the sample programs?

A1. Using gdb, if the global variables (i.e the mdata section) do not get loaded into the simulator memory simulating the ROM, compile the program without -DROMSTART option in the makefile. This will ensure that the code to copy ROM image of data into the RAM in start.s file is not executed.

This particular code is required only for running the code on the hardware. When you are using the GDB, this code fragment is not required.

When using GDB, "start.s" file can also be removed from the project and the default crt0.o can be used (by removing -nostartfiles option) while building the project.

Q2. How should I use scanf() to get the input from the console window of a debugger?

A2. Use the command line debugger e.g. sh-elf-gdb to get the input using scanf().

If you use Insight GUI, you will not be able to get the input from the console window because Insight does not support target input.

You can also run Insight with -nw switch ( No window interface ) to use scanf().

Q3. How do I install the GDBSH Debugger/Simulator on a GNU/Linux machine?

A3. There is one RPM provided for installing GDBSH Debugger/Simulator on a GNU/Linux machine.

An rpm is a specially formatted file, which has all the files, documentation and configuration files. For each object file format, for example ELF of GDBSH v0304, one binary rpm or package file is provided. The name of this rpm is prefixed with i386.rpm. This rpm is used to install the package on the machine.

NOTE: To install the rpm on your machine you should have root access to the rpm database.

For example, to install GDBSH v0304 on Linux using rpm, on terminal window type:
rpm-ivh +
e.g. rpm -ivh GDBSH_v0304_ELF-1-1.i386.rpm

This will install GDBSH_v0304 elf package if one is installing it for the first time. By default it installs it in /usr directory. To change the default directory use --prefix install mode option.

To upgrade the package use following command:
rpm -Uvh +
The upgrade command can also be used even if it is the first install.

To uninstall the package
rpm -e +
e.g. rpm -e GDBSH_v0304_ELF
Where GDBSH_v0304_ELF is the name of the package.

To install or uninstall from X-Windows open the package from RPM-Manager and click on install or uninstall tab or click the rpm file in the file manage

For non-privileged users (i.e. other than root):

You should have rights on the rpm database to install this rpm. The install and upgrade options to use are same as mentioned for privileged users.

For more information refer man page of rpm or visit http://www.rpm.org/max-rpm
