Q1. How do I generate S-Record MOT file automatically from HEW ?

A1.You can achieve this by two methods.

1. The linker may be used to generate a *.mot file
Go to the linker menu of the "Options" Menu.
Go to the "output" page of the linker settings dialog box.
From the "Output format" drop-down list select "Generate S-Record file"
as the output format.

However, with this you will lose all the debugging information as the *.mot file does not contain the debugging info.

2. The *.mot file can also be generated using the objcopy utility.
To achieve this you can specify an extra custom phase in HEW itself so as to generate *.mot file along with the*.out file.

Follow the steps given below.

a. Go to the "Build Phases" option from "options" menu.

b. Click on Add. Select "Create a new custom phase" and click "Next".

c. Select "Linker Output file" as Input file type, "Single phase" and click "Next"

d. Set "Command" as

"$(HEWDIR)\Tools\KPIT Cummins\GNUSH-COFF\v0304\Sh-coff\bin\sh-coff-objcopy.exe",
"Default options" as
"-O srec $(CONFIGDIR)\$(PROJECTNAME).x $(CONFIGDIR)\$(PROJECTNAME).mot" and click "Next"

e. Click "Finish"

f. Check "Phase" option and it should be the last ie after linker, click "Ok"

g. Now when you will choose "Build All" from the "Build" menu, it will build the .mot file in your configuration directory.

Q3. I am unable to source-level debug HEW generated project for COFF tool chain. Why ?

A3.HEW creates the output file in the debug folder of the project workspace.

This means that the source files and the generated output file are in different folders.

The coff debug format does not have directory information.

You need to set directories to be browsed for source using directory command from the console window.

You may view the current directory by using "pwd" command.

e.g. If the project named sh1 is created in folder c:\tmp.

Then the source programs will be in folder is C:\tmp\sh1\sh1\ and the executable will be in c:\tmp\sh1\sh1\debug.

In such caseload the executable from debug folder.

Open the command window. And type following command.
(gdb) pwd

This will show current directory as "/cygdrive/c/tmp/sh1/sh1/debug"
(gdb) directory/cygdrive/c/tmp/sh1/sh1

With this setting, you will be able to source-level debug the program.

Q4. How do I manipulate heap and stack through HEW ?


A4.To change the location of the heap in HEW using _end symbol, on the toolbar go to Options and then select Linker.

It will display the Linker options dialog box.

In the Sections tab page you can add a new section by clicking on the Add group button.

In the new section dialog box you can type a name say .heap_start which can have start address as a fixed address.

Delete the _end symbol from the previous section and add in this new section.

To manipulate the size of heap and stack copy sbrk.c file form \newlib\libc\sys\h8300hms to the HEW project and modify the same.The building of changed sbrk.c along with the HEW project avoids rebuilding of the newlib sources.

Q5. How do I pre-process assembly files in HEW ?

A5.    HEW treats *.s and *.S files in same way. Because of this the '*.s' files are not preprocessed and are given directly to the assembler.

To pre-process assembly files, please use file extension "asm". So all files with extension

"asm" will be pre-processed by HEW ( using GCC) and then passed to the assembler.

Q6. How do I debug binaries created by ELF toolchain ?

A6.GNUH8 v0303 ships with the latest ELF/DWARF-2 Object reader suitable for HEW 2.x. GNUSH/H8 v0602 ships with the latest ELF/DWARF-2 Object reader suitable for HEW '4.x' only.

To use this object reader go to the menu "Debug Settings" from "Options".

From the dialog box, using "Default Debug Format" drop-down list, select "ELF/DWARF2 for KPIT".

Using this you should be able to debug the binaries using ELF/DWARF2 debug format.

Before downloading the binary file please configure memory map for the simulator which is under "Options->Simulator->Memory Resource" menu.

Note:

1. This object reader will work only with HEW version 2.x.
2. If you select "ELF/DWARF2" instead of "ELF/DWARF2 for KPIT", the debugging may not work and HEW may crash.

Q8. Are there any issues when more than one versions of HEW installed on the same machine ?

A8.Yes. User may face problems while registering the KPIT Cummins GNU tool chain with HEW where more than one installation is v2.1 or earlier.

To avoid this problem, we have modified installers.

If HEW 2.1 and HEW 1.3 are installed on the machine, then the GNUH8 / GNUSH installer will install the tool chain only for one version of HEW selected by the user.

To install tool chain for other version of HEW, user will have to install the tool chain again. During second installation please go through "Modify" installation option and select the other version of HEW.

Besides this, at the end of installation, the installer will ask user if user wants to start HEW in order to register the tool chain automatically.

If user selects "yes", then the installer will automatically invoke the HEW and register the GNUSH / GNUH8 tool chain.

If the user selects "no", then user must start correct version of HEW (selected while installing the tool chain ) first.

If the user start the other version first, then the tool chain will not get registered properly.

e.g. User selects to install tool chain for HEW1.3 and selects not to start HEW automatically.
In such case if user starts HEW 2.1 before HEW 1.3, the tool chain will get registered to HEW 2.1 and not in HEW 1.3.

Note: Toolchain integration with HEW-1.x and HEW-3.x is discontinued from v0602 release. v0602 onwards, GNUSH / GNUH8 / GNUM16CM32C toolchains will get integrated only with HEW-4.x (or later updates of HEW). However, free technical support for older toolchain version with HEW-1.x and HEW-3.x will be continued.

Q9. How to setup the simulator to debug my code in HEW ? or What do I do when getting "memory access error" while using HEW simulator?

A9.To use the simulator shipped with HEW, first you need to configure the Simulator.

To configure the Simulator follow the steps given below. The project should be built before this stage.

1. Select "Debug Settings" from Options menu.

2. This will display a dialog box with two tabs "Target" and "Options".

3. Using the "Target" drop-down list, select the appropriate CPU series for your project.
e.g. H8/300, H8/300H, H8S/2000A or SH1,SH2 etc.

4. From "Default Debug Format" drop-down list, select option "ELF/Dwarf2 for KPIT".

5. Add the generated output file (normally $projectname.x from debug folder) using the download module button.

6. Clicking on "Add" button, will display another dialog box named "Download Module".

7. Let the offset be 0x0. Select "Format" as "ELF/Dwarf2 for KPIT" and select the generated output file using the browse button.

8. Press OK to confirm your settings. This will add various tool bars to the HEW menu.

9. This will also add the *.x file in project workspace under "Download Module".

10. Select on "Simulator" option from "Options" menu and select "Memory resource".

11. Click on the add button and enter memory details of your target. After this simulator will allocate memory required for the project.
After this simulator will allocate memory required for the project.

12. Right click on the *.x file under "download module" in project workspace window, to download the file to the simulator. After this the user can use simulator to debug the application downloaded.

In case of HEW4.x there is no need to allocate the memory resources since the HEW4 simulator automatically allocates the memory resources depending upon the dowloadable segments in the output file. However, the stack has no download segment associated with it, hence it doesn't automatically get allocated and must be manually allocated by the user. If the memory region is not allocated for the stack, the user will get "Memory Access Error" while debugging.

Q15. On adding Memory region a warning "no memory region specified for section `.bss' " is encountered. How can this error be ignored through HEW?

A15. This warning occurs if no memory region is specified for that section. This warning can be safely ignored. The default project generated through HEW has a section .bss with data in it but no memory region specified. To remove this warning assign the memory region for that section.
e.g. Suppose the memory section RAM is added, then in the linker sections dialog box modify the settings of section for which the warning is generated as follows:

    Start Address: Memory Region

    Region: RAM

Q16. Why do I need to manually uninstall the KPIT toolchain(s) from HEW and how do I do that?

A16. When the KPIT toolchain is uninstalled from the start menu, it does not get uninstalled from HEW. The user will have to manually uninstall the toolchain from HEW.

Please follow the steps below to uninstall the KPIT toolchain from HEW4.04 or lower versions,

1. Close all your workspaces in HEW. A toolchain cannot be uninstalled from HEW if a workspace is open.

2. Go to "Tools->Administration". The toolchain first needs to be unregistered before uninstallation.

3. In the "Tools Administration" window, under "Registered Components-> Components-> Toolchains", select the toolchain that you want to uninstall.

4. Click on "Unregister" button. A dialog box will pop up asking if you want to unregister the tool, click on "Yes".

5. In the "Tools Administration" window, click on the button "Uninstaller". The "Uninstall HEW Tool" window will pop up.

6. Click on "Start" button, this will display all the tools in HEW that can be uninstalled.

7. Select the toolchain that you want to uninstall (the toolchain that you unregistered in step 4) and then click on the button "Uninstall".

8. A "confirmation" dialog box will pop up, displaying all the files that are present in the toolchain and ask you if you want to delete all the above files & folders. Click "OK" button. The toolchain will be uninstalled from HEW.

9. If some user made files or folders are present in the toolchain directory, a dialog box will pop up saying that "Unable to uninstall, the directory in not empty". In this case click "Ignore" button and later delete those files from the toolchain directory.

10. Click the "close" button on the "Uninstall HEW Tool" window. Click on the "OK" button in "Tools Administration" window.

The above mentioned "Uninstaller" feature is not available in case of HEW4.05. The toolchain will have to be manually deleted from the directory, "$(HEWINSTALLDIR)ToolsKPIT CumminsGNUXXX-ELFvXXX" where,
$(HEWINSTALLDIR) is the HEW4.05 installation directory (e.g. "C:Program FilesRenesasHew"), GNUXXX-ELF is the toolchain to be uninstalled (e.g. "GNUH8-ELF") and vXXX is the toolchain version to be uninstalled (e.g. "v0901").

Q 17. After upgrading a project to v0902, why are included assembly files not found?

A 17. Till v0901, the library generator, compiler, assembler and linker build phases had separate GUI interfaces in HEW. These separate GUI interfaces in HEW have been replaced by a single interface to allow users to make application-wide changes far more easily.

However, this single interface requires to add the path of include files while building the project.

Thus to fix this problem, after upgrading the project to v0902, add the path of the included assembly file under:
Build-> KPIT GNUH8[ELF] Toolchain-> Assembly-> Category : Source -> Add include file directory -> .

Q.21. How do I use the E8 emulator along with the KPITGNUM16CM32C toolchain to debug my application?

A.21. After installing the GNUM16CM32C toolchain using the latest installer, follow the steps given below,

1. Connect the hardware to the USB port through the E8 emulator. The LED on the E8 emulator will glow indicating that it is connected properly.

2. Start High Performance Embedded Workshop (HEW). Select "Refer to another project workspace". Click "OK". Open the folder containing your application and select ".hws" file and click "Open". Ensure that the application is in

3. Go to "Debug" option in the menu. Click on the "Debug Settings" option. A window for debug settings will appear. Select "R8C E8 System" from the target dropdown list. Select "Elf/Dwarf2_KPIT" from the "Debug Download Format" dropdown list. Click on the "Add" button. Select "Elf/Dwarf2_KPIT" from the "Format" dropdown list. 6. Check on the "Power supply is carried out" checkbox. Select power supply of 5V. Click "OK".

8. Put the break points if required.

9. Go to "Debug", click on "Reset CPU" and click "Go".

10. The program starts running and halts at the break point. You can debug your application now.


Limitation: The pointers and local variables cannot be watched in HEW.

Q24. In HEW, on using the optimized libraries with C++ project, why does the linker generate errors of "undefined references" to some routines while building the project?

A24. As it is mentioned in the release notes, some of the ANSI C library functions are not implemented in H8 optimized libraries. The library "libstdc++.a" is dependant on C library for some of the routines like "_fflush", "_lseek", "_strdup" etc. As these are not implemented in optimized libraries linker generates the "undefined reference" errors.

Q 25. After enabling rules of "Static Analysis" in HEW, why do i get compiler error "unrecognized command line option '-Wanalyze-Rule1'"?

A 25. Support for "Static Analysis" of code is added in HEW for GNUSH v0603 toolchain. However, the command line options which are used in HEW to support this feature are incorrect. The option "-Wanalyze-Rule1" is used in HEW instead of "-Wanalyze-rule=1". Similarly the options for all the other rules are also incorrect. Since the command line options are unrecognized, you get the compilation error.

Please download and install the following file to fix this problem:
GNUSHv0603-ELF-MP1.exe

Please note that this fix is required only for GNUSH-v0603 toolchain.

Q 26. What does the 'Library Generator' phase do?

A 26. The 'Library Generator' phase supports the library generator tool, 'libgen'.
This tool builds the standard/optimized libraries with customized compiler and assembler options.
Until now, only the pre-built libraries were provided with the toolchain. The ‘libgen’ tool can be used on the command line as well as in HEW.

'Library Generator' on the Command Line:
For information on the usage of the 'libgen' tool and the supported options, please refer to the following link,
http://www.kpitgnutools.com/manuals/binutils.html#SEC15

'Library Generator' in HEW:
A new phase "Library Generator" is added to interface the 'libgen' tool in HEW.
The settings for this phase are as follows,
a. The "Settings" tab provides the options to select the type of the library i.e. 'Pre-Built' or 'Project-Built', the source i.e. 'Newlib' or 'Optimized' and the mode used for building the library.
b. The "Header files" tab provides the option to select the standard header files for building the library.
c. The "Other options" tab allows the user defined options to be passed to the "Library Generator" phase.

Note: Currently, after changing the 'Library Generator' settings, the user is prompted to open the Compiler/Linker option dialogs and click on 'OK'. We are aware that this is inconvenient for the users and we are working on a fix for the same.

Q 27. What are the known limitations and issues in the 'Library Generator' phase?

A 27. The known limitations and issues in the 'Library Generator' phase are,
1. Irrespective of the type of the source file, the 'Library Generator' options dialog is invoked if we right click on the source file in the 'Workspace' window and select
'Build -> KPIT GNUXXX [ELF] Toolchain -> C/C++' or 'Build -> KPIT GNUXXX [ELF] Toolchain -> Assembly'.

2. If the target specific options are selected under 'C/C++ -> CPU -> Additional CPU options', these options are reflected in the 'Library Generator' settings as well.
However, if the target specific options are passed under 'C/C++ -> Other -> User defined options', these options are not reflected in the 'Library Generator' settings. Hence, these options have to be specified again in the 'Library Generator' phase under
'Other options'->'User defined compiler options'.

Q 28. While creating a project for M16C or M16C/80, M32C, I get an error message "An illegal character is used in the directory name, use only letters, digits or '_'". Also, after clicking 'OK', the dialog box 'Select Target CPU' is displayed, but the 'CPU Series' and 'CPU Group' list boxes are empty. I am not able to proceed further with the project generation. How can I avoid this problem?

A 28. The error message is displayed when the project directory path contains spaces. Please create the project in a path that does not contain any spaces e.g. 'C:\Workspace\'.

You can also set the default directory to a specific location (e.g. 'C:\Workspaces') by following the steps given below:
1. Select the [Setup->Options...] menu option. The Setup Options dialog will be displayed. Select the 'Workspace' tab.

Q 29. What are the known limitations in the upgraded build phase interfaces?

A 29. Till v0901, the library generator, compiler, assembler and linker build phases had separate GUI interfaces in HEW.

Q 30. Why do the "Miscellaneous" options selected in GNUH8 and GNUM16CM32C assembler phase not remain selected after upgrading project to v0902 or higher versions?

A 30. Following options are provided as "Miscellaneous Options" in the Assembler phase of GNUH8 and GNUM16CM32C projects:
1. Suppress warning messages
2. Include local labels
3. Issue warnings when difference tables altered
4. Work faster (assembly from trusted compiler)
5. Display assembly statistics

When some of these options are selected in projects created with v0901 or lower toolchain versions, the selection is not maintained when the projects are upgraded to v0902 or higher versions.

Q31. What are the differences between 'optlib' and 'newlib' libraries, and 'Project-Built' and 'Pre-Built' libraries?
A 31. User has the option to use either standard GNU ('newlib') libraries or optimized ('optlib') libraries.

'newlib' - The GNU 'newlib' libraries are complete ISO C/C++ libraries but are relatively costly in terms of code size for smaller embedded architectures.

'optlib' - The KPIT 'optlib' libraries are specially optimized libraries that have been custom-written for embedded applications but supports a sub-set of the ISO C library and does not support the C++ library.
The optimized libraries ('liboptm.a' and 'liboptc.a') are not provided under GNU GPL. The source code of these optimized libraries is neither released nor available on request.

'Project-built' - On selecting the 'Project-built' setting in HEW, libraries will be custom-built with your normal project build based on your specific options (e.g. a particular type of compiler optimization, assembler optimization).

'Pre-built' - On selecting the 'Pre-built' setting in HEW, the complete libraries will be used that are supplied as part of the KPIT toolchain.

Q32. What are the differences between the sh2a, sh2a (nofpu), sh2a (single prec only) and sh2a (single prec default) of the SH target series?
A 32. sh2a - '-m2a': Generates code for the SH2a target. The operations are performed using FPU in double precision mode.

sh2a (nofpu) - '-m2a-nofpu': Generates code for the SH2a without a floating-point unit.
i.e., The compiler does not generate FPU instructions. Therefore every float or double operation will be made with normal asm instructions or using math library calls. The compiler will generate code for float and double operations based on the data type of the variable being used when the '-m2a-nofpu' option is used.

sh2a (single prec only) - '-m2a-single-only': Generates code for the SH2a with a floating-point unit that only supports single-precision arithmetic. When the '-m2a-single-only' (sh2a - single prec only) option is selected, FPU is used only for float operations (single precision instructions only) and will not support operations in double precision mode. In this case compiler will generate optimized code as it does not have to take care of the mode switching between single and double precision.

sh2a (single prec default) - '-m2a-single': Generates code for the SH2a assuming the floating-point unit is in single-precision mode by default. Depending on the data types used, it will also support FPU unit in double-precision mode.

The following libraries are selected in HEW depending on the target selection,

Library Used - m2

Target Variant - SH2A-SINGLE-ONLY
Compiler Option - m2a-single-only
Library Used - m2e

Target Variant - SH2A-SINGLE

Q1. Why HEW crashes while debugging GNU HMON tutorial/application project in following scenario-
GNU toolchain - v0502, v0503.
Optimization - O1, O2, O3, Os.
The project works fine with same optimization option if built and debugged with earlier GNU toolchain.?

A1. GNU toolchains v0502 and v0503 are based on GCC 4.0. The optimization option "-fmerge-constants" in GCC 4.0 incorrectly optimizes some constants used by HMON. This incorrect optimization in HMON code results in HEW crash. This option is passed at all optimization levels. The workaround to this problem is to pass the option "-fno-merge-constants" alongwith the optimization option.
